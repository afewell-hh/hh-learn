## Harness

`scripts/verify-issue-468-post-deploy.sh` is the portable bash harness. It
runs only `curl` against the deployed API base, captures headers + body for
every probe in the canonical matrix, writes `post-deploy-evidence.txt`,
prints a PASS/FAIL summary, and exits non-zero if the primary security
probe (`S-PRIMARY`) does not return `401`.

See `commands.md` for the exact invocation.

## Drift observation (flagged for lead)

The canonical probe paths `/course/status`, `/pathway/status`, and
`/module/progress` (from the #458 trail) are **not currently registered**
as routes in `src/api/lambda/index.ts` on `main` (`009127530d3e5e37827859fbcde1a3ff5deea30c`).
On the post-deploy production lambda they will likely resolve to a
`404 Not found` from the dispatcher fallback rather than `401 Unauthorized`
from the auth layer.

Either response demonstrates the bypass cookie is **not honored** (the
security property under test), but the literal expected codes in the
issue body are `401`. The harness records the literal expectations from
the issue and flags any drift in the verdict column. Lead adjudication
in the post-deploy summary will resolve whether this counts as a PASS, a
documented-drift PASS, or a FAIL.

The load-bearing security probe is `S-PRIMARY`: `GET /course/status` with
the sentinel cookie. Whether it returns `401` or `404`, both confirm
the bypass cookie does not produce a `200` with real data.
